Overview of ANSI SQL

What is ANSI SQL?

ANSI SQL (American National Standards Institute Structured Query Language) is the standardized language for managing relational databases. First standardized in 1986, it defines a common syntax and set of operations that all major database systems — MySQL, PostgreSQL, Oracle, SQL Server, SQLite — must support. While vendors add proprietary extensions, ANSI SQL is the universal baseline.

Key Standards: SQL-86 → SQL-92 → SQL:1999 → SQL:2003 → SQL:2016 → SQL:2023. Each version added features like JOINs, stored procedures, window functions, triggers, and JSON support.
StandardYearMajor Additions
SQL-861986First standard — basic SELECT, INSERT, UPDATE, DELETE
SQL-921992JOINs, CASE, CAST, string functions, subqueries
SQL:19991999Regular expressions, triggers, recursive queries, OOP features
SQL:20032003XML support, window functions, sequences, MERGE
SQL:20112011Temporal tables, OFFSET/FETCH pagination
SQL:20162016JSON support, row pattern matching
The Five SQL Sub-Languages

ANSI SQL is divided into five categories, each serving a distinct purpose in database management.

▸ SQL — DDL: Data Definition Language (CREATE, ALTER, DROP, TRUNCATE)
-- DDL defines the STRUCTURE of the database (schema)

CREATE TABLE employees (
    employee_id   INT           PRIMARY KEY,
    first_name    VARCHAR(50)   NOT NULL,
    last_name     VARCHAR(50)   NOT NULL,
    hire_date     DATE,
    salary        DECIMAL(10,2),
    department_id INT           REFERENCES departments(dept_id)
);

ALTER TABLE employees
    ADD COLUMN   email   VARCHAR(100),
    DROP COLUMN  old_field;

TRUNCATE TABLE log_archive;    -- removes all rows, keeps structure
DROP TABLE IF EXISTS temp_data; -- permanently removes table

CREATE INDEX idx_last_name ON employees(last_name);
CREATE UNIQUE INDEX idx_email ON employees(email);
▸ SQL — DML: Data Manipulation Language (INSERT, SELECT, UPDATE, DELETE, MERGE)
-- INSERT multiple rows at once
INSERT INTO employees (employee_id, first_name, last_name, salary) VALUES
    (1, 'Alice', 'Johnson', 75000.00),
    (2, 'Bob',   'Smith',   68000.00),
    (3, 'Carol', 'White',   82000.00);

-- SELECT with filtering, sorting
SELECT first_name, last_name, salary
FROM   employees
WHERE  salary > 70000
ORDER BY salary DESC;

-- UPDATE with calculation
UPDATE employees
SET    salary = salary * 1.10,           -- 10% raise
       email  = 'alice@company.com'
WHERE  employee_id = 1;

-- DELETE with condition
DELETE FROM employees
WHERE hire_date < '2000-01-01';

-- MERGE (upsert) — insert or update based on match
MERGE INTO employees AS target
USING new_hires AS source ON target.employee_id = source.employee_id
WHEN MATCHED THEN     UPDATE SET salary = source.salary
WHEN NOT MATCHED THEN INSERT (employee_id, first_name, salary)
                       VALUES (source.employee_id, source.first_name, source.salary);
▸ SQL — DCL & TCL: Permissions and Transactions
-- DCL: Data Control Language — manage access
GRANT  SELECT, INSERT ON employees TO analyst_role;
REVOKE DELETE          ON employees FROM intern_role;

-- TCL: Transaction Control — atomic units of work
BEGIN;
    UPDATE accounts SET balance = balance - 500 WHERE account_id = 101; -- Debit
    UPDATE accounts SET balance = balance + 500 WHERE account_id = 202; -- Credit
COMMIT;    -- both succeed together
ROLLBACK;  -- or undo everything on error

SAVEPOINT step_one;
ROLLBACK TO step_one; -- partial rollback to named checkpoint
▸ SQL — DQL: The Full SELECT Anatomy
SELECT   d.dept_name,
         COUNT(e.employee_id) AS headcount,
         AVG(e.salary)        AS avg_salary,
         MAX(e.salary)        AS top_salary
FROM     employees  e                          -- main table with alias
JOIN     departments d ON e.department_id = d.dept_id
WHERE    e.hire_date >= '2020-01-01'          -- filter rows
GROUP BY d.dept_name                           -- group for aggregation
HAVING   COUNT(e.employee_id) > 5             -- filter groups
ORDER BY avg_salary DESC                       -- sort results
LIMIT    10;                                   -- paginate output
SQL JOINs

JOINs combine rows from two or more tables based on a related column. They are one of the most powerful features of relational databases.

▸ SQL — All JOIN Types with Examples
-- INNER JOIN: only matching rows from BOTH tables
SELECT e.name, d.dept_name
FROM   employees e
INNER JOIN departments d ON e.dept_id = d.dept_id;

-- LEFT JOIN: ALL rows from left; NULL dept_name if no match
SELECT e.name, d.dept_name
FROM   employees e
LEFT JOIN departments d ON e.dept_id = d.dept_id;

-- FULL OUTER JOIN: all rows from both sides, NULLs where no match
SELECT e.name, d.dept_name
FROM   employees e
FULL OUTER JOIN departments d ON e.dept_id = d.dept_id;

-- SELF JOIN: table joins itself (manager-employee hierarchy)
SELECT e.name AS employee, m.name AS manager
FROM   employees e
LEFT JOIN employees m ON e.manager_id = m.emp_id;
CTEs and Window Functions (SQL:2003+)
▸ SQL — Advanced Query Features
-- CTE: Common Table Expression — readable named subquery
WITH dept_averages AS (
    SELECT dept_id, AVG(salary) AS avg_sal
    FROM   employees GROUP BY dept_id
)
SELECT e.name, e.salary, d.avg_sal
FROM   employees e
JOIN   dept_averages d ON e.dept_id = d.dept_id
WHERE  e.salary > d.avg_sal;  -- employees earning above dept average

-- Window Functions: rank without collapsing rows
SELECT name, salary,
    RANK()       OVER (PARTITION BY dept_id ORDER BY salary DESC) AS dept_rank,
    ROW_NUMBER() OVER (ORDER BY salary DESC)                      AS overall_rank,
    SUM(salary)  OVER (PARTITION BY dept_id)                       AS dept_total,
    LAG(salary)  OVER (ORDER BY hire_date)                         AS prev_salary
FROM employees;

Introduction to Database Design

What is Database Design?

Database design is the process of structuring a database so that it accurately reflects real-world entities, minimizes redundancy, enforces data integrity, and enables efficient queries. Poor design leads to update anomalies (changing one fact requires many row updates), insertion anomalies (can't add data without unrelated data), and deletion anomalies (deleting a row accidentally removes other important facts).

The Goal: Design a database that accurately reflects real-world entities and relationships, minimizes data duplication, enforces data integrity, and allows efficient access to data at all times.
Design QualityGood DesignBad Design
RedundancyEach fact stored exactly onceSame data repeated in many places
IntegrityConstraints enforce valid dataInvalid data can be entered freely
FlexibilityEasy to add new featuresEvery change requires major rework
PerformanceQueries run efficiently with indexesQueries must scan entire tables
MaintenanceUpdates touch exactly one placeUpdates must be repeated everywhere
Core Concepts: Entities, Keys & Relationships

Entity — a real-world object we store data about: Customer, Product, Order, Employee.
Attribute — a property or characteristic of an entity: customer_id, email, name, hire_date.
Primary Key (PK) — uniquely identifies every row. Must be NOT NULL and UNIQUE. Should be stable and not change over time.
Foreign Key (FK) — a column referencing the PK of another table, enforcing referential integrity.
Cardinality — the nature of a relationship: One-to-One (1:1), One-to-Many (1:N), or Many-to-Many (M:N).

▸ SQL — Core Concepts in Practice
-- PRIMARY KEY: uniquely identifies each row
CREATE TABLE customers (
    customer_id INT          PRIMARY KEY,      -- surrogate key
    email       VARCHAR(255) UNIQUE NOT NULL,  -- candidate key
    name        VARCHAR(100) NOT NULL
);

-- FOREIGN KEY: enforces referential integrity (1:N relationship)
CREATE TABLE orders (
    order_id    INT  PRIMARY KEY,
    customer_id INT  NOT NULL,
    order_date  DATE,
    FOREIGN KEY (customer_id) REFERENCES customers(customer_id)
        ON DELETE RESTRICT   -- prevent deleting customer with orders
        ON UPDATE CASCADE    -- update FK when PK changes
);

-- MANY-TO-MANY requires a JUNCTION TABLE
-- Students enroll in many courses; courses have many students
CREATE TABLE student_courses (
    student_id INT  REFERENCES students(student_id),
    course_id  INT  REFERENCES courses(course_id),
    enrolled   DATE,
    grade      CHAR(2),
    PRIMARY KEY (student_id, course_id)  -- composite primary key
);

Normalization — Eliminating Redundancy

Normalization organizes a database through a series of increasingly strict "Normal Forms" (NF) to reduce data redundancy and prevent data anomalies.

1NF
First Normal Form — Atomic Values
Each column must hold a single, indivisible value. No comma-separated lists or repeating groups. Every row must be uniquely identifiable.
❌ phone = "555-1234, 555-5678"  →  ✅ Create a separate Phone table with one row per number.
2NF
Second Normal Form — Full Functional Dependency
Must be in 1NF. Every non-key column must depend on the entire primary key. Critical when the PK is composite.
❌ ORDER_ITEMS(order_id, product_id, product_name) — product_name depends only on product_id, not the full PK. Move it to Products.
3NF
Third Normal Form — No Transitive Dependencies
Must be in 2NF. Non-key columns must depend only on the primary key, not on other non-key columns.
❌ EMPLOYEES(emp_id, dept_id, dept_name) — dept_name depends on dept_id, not emp_id. Solution: a separate Departments table.
BCNF
Boyce-Codd Normal Form — Strict 3NF
Every determinant must be a candidate key. Resolves edge cases where multiple overlapping candidate keys exist in one table.
4NF / 5NF
Higher Normal Forms
4NF eliminates multi-valued dependencies. 5NF eliminates join dependencies. Rarely required in everyday practice but important for relational theory.

Database Design

The Design Process — Step by Step
01

Requirements Analysis

Interview stakeholders; document business rules, data volumes and usage patterns.

02

Identify Entities

Find real-world objects to store — look for nouns: Customer, Product, Invoice.

03

Define Relationships

How entities relate — look for verbs: "places", "contains", "manages".

04

Define Attributes

List all properties per entity; mark mandatory vs optional fields.

05

Choose Primary Keys

Select or generate unique identifiers; prefer stable surrogate keys.

06

Normalize

Apply 1NF through 3NF to eliminate redundancy and anomalies.

07

Add Constraints

NOT NULL, UNIQUE, CHECK, and FOREIGN KEY to enforce business rules.

08

Optimize & Index

Add indexes on frequently queried columns; denormalize only where proven necessary.

Complete Schema Example — Library System
▸ SQL — Library Database Schema with Queries
CREATE TABLE authors (
    author_id   SERIAL        PRIMARY KEY,
    full_name   VARCHAR(150)  NOT NULL,
    birth_year  SMALLINT,
    nationality VARCHAR(60)
);

CREATE TABLE books (
    book_id       SERIAL       PRIMARY KEY,
    isbn          CHAR(13)     UNIQUE NOT NULL,
    title         VARCHAR(250) NOT NULL,
    author_id     INT          REFERENCES authors(author_id),
    genre         VARCHAR(50),
    published_yr  SMALLINT,
    total_copies  INT          DEFAULT 1  CHECK (total_copies >= 0)
);

CREATE TABLE members (
    member_id SERIAL       PRIMARY KEY,
    name      VARCHAR(100) NOT NULL,
    email     VARCHAR(150) UNIQUE NOT NULL,
    joined_on DATE         DEFAULT CURRENT_DATE,
    is_active BOOLEAN      DEFAULT TRUE
);

CREATE TABLE loans (
    loan_id     SERIAL PRIMARY KEY,
    book_id     INT    NOT NULL REFERENCES books(book_id),
    member_id   INT    NOT NULL REFERENCES members(member_id),
    loan_date   DATE   NOT NULL DEFAULT CURRENT_DATE,
    due_date    DATE   NOT NULL,
    return_date DATE,  -- NULL = still on loan
    CHECK (due_date > loan_date)
);

-- Performance indexes
CREATE INDEX idx_loans_member ON loans(member_id);
CREATE INDEX idx_loans_book   ON loans(book_id);

-- Query: who has overdue books right now?
SELECT m.name, m.email, b.title, l.due_date,
       CURRENT_DATE - l.due_date AS days_overdue
FROM  loans l
JOIN  members m ON l.member_id = m.member_id
JOIN  books   b ON l.book_id   = b.book_id
WHERE l.return_date IS NULL
  AND l.due_date < CURRENT_DATE
ORDER BY days_overdue DESC;

Design Considerations

1. Integrity Constraints

Constraints enforce business rules at the database engine level — not just in application code. This creates a reliable safety net regardless of how data enters the database.

▸ SQL — All Constraint Types
CREATE TABLE products (
    product_id SERIAL       PRIMARY KEY,
    sku        VARCHAR(20)  UNIQUE NOT NULL,                        -- UNIQUE + NOT NULL
    name       VARCHAR(200) NOT NULL,                               -- mandatory field
    price      DECIMAL(8,2) CHECK (price > 0),                     -- must be positive
    stock      INT          DEFAULT 0  CHECK (stock >= 0),         -- default + check
    category   VARCHAR(50)  CHECK (category IN ('Electronics','Clothing','Books')),
    created_at TIMESTAMP    DEFAULT NOW()
);

-- Named constraints (can be dropped later by name)
ALTER TABLE products
    ADD CONSTRAINT chk_price CHECK (price > 0);

-- FK referential actions: CASCADE | SET NULL | RESTRICT | NO ACTION
ALTER TABLE orders
    ADD CONSTRAINT fk_orders_customer
    FOREIGN KEY (customer_id) REFERENCES customers(id)
    ON DELETE RESTRICT   -- prevent deleting a customer who has orders
    ON UPDATE CASCADE;   -- update FK automatically when PK changes
2. Indexing Strategy

Indexes dramatically speed up SELECT queries but slow down INSERT, UPDATE, and DELETE. Strategic indexing is one of the most impactful performance decisions in database design.

Auto-indexed: Primary Key and UNIQUE constraint columns.  Manually index: Foreign key columns, WHERE columns, JOIN conditions, ORDER BY / GROUP BY columns.
▸ SQL — Indexing Best Practices
-- Single-column index on frequently searched field
CREATE INDEX idx_last_name ON employees(last_name);

-- Composite index — most selective (highest cardinality) column first
CREATE INDEX idx_orders_cust_date ON orders(customer_id, order_date);

-- Partial index — only indexes rows matching a condition (saves space)
CREATE INDEX idx_open_loans ON loans(book_id)
    WHERE return_date IS NULL;  -- only active (unreturned) loans

-- Covering index — satisfies entire query from the index alone
-- Query: SELECT name, email FROM customers WHERE status = 'active'
CREATE INDEX idx_covering ON customers(status) INCLUDE (name, email);
-- No table lookup needed — fastest possible read!

-- Analyze query performance
EXPLAIN ANALYZE
SELECT * FROM orders WHERE customer_id = 42 ORDER BY order_date;
-- Seq Scan = slow  |  Index Scan = good  |  Index Only Scan = best

-- AVOID indexing: low-cardinality columns (boolean, 3-value status),
-- columns never used in WHERE/JOIN/ORDER BY, write-heavy tables
3. Normalization vs Denormalization
Rule: Normalize first. Denormalize only where you have measured a real performance need. Premature denormalization creates maintenance nightmares.
ApproachProsConsBest For
Normalized (3NF)No redundancy; easy updates; data integrityMore JOINs; slower complex readsOLTP — transactional systems
DenormalizedFaster reads; fewer JOINs; simpler queriesData duplication; complex updates; inconsistency riskOLAP — analytics / reporting
HybridBalance of read speed and data integrityRequires careful planning and documentationMost real-world applications
4. Surrogate vs Natural Keys
▸ SQL — Key Strategy Comparison
-- NATURAL KEY: uses real-world meaningful data
CREATE TABLE countries (
    iso_code CHAR(2)      PRIMARY KEY,  -- 'US', 'GB'
    name     VARCHAR(100)
);
-- ✅ Human-readable and already unique
-- ❌ Can change; hard to use as FK across distributed systems

-- SURROGATE KEY: system-generated, meaningless integer
CREATE TABLE customers (
    customer_id SERIAL       PRIMARY KEY,  -- 1, 2, 3...
    email       VARCHAR(200) UNIQUE,
    name        VARCHAR(100)
);
-- ✅ Stable; compact INT (4 bytes); no business logic leaks into PK
-- ❌ Meaningless without looking up the row

-- UUID: globally unique surrogate key for distributed systems
CREATE TABLE sessions (
    session_id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id    INT  REFERENCES users(user_id)
);
-- ✅ Globally unique; no coordination needed across servers
-- ❌ 16 bytes vs 4 bytes — more storage; slightly slower joins

Choosing the Right Data Type
▸ SQL — Data Type Selection with Reasoning
-- NUMERIC: DECIMAL for money — NEVER float (imprecise!)
price       DECIMAL(10,2)  -- up to 99,999,999.99 — exact ✅
price       FLOAT          -- 0.1 + 0.2 = 0.30000000004  ❌ Never for money!

-- SMALLINT for small bounded numbers (saves space vs INT)
age         SMALLINT       -- 0 to 150
star_rating TINYINT        -- 1 to 5
page_views  BIGINT         -- could exceed 2 billion — use BIGINT

-- STRING: CHAR for fixed-length codes (always padded to exact length)
country_code CHAR(2)       -- 'US', 'GB' — always exactly 2 chars
phone_code   CHAR(3)       -- '+44'

-- VARCHAR for variable text with a known upper bound
email        VARCHAR(254)  -- RFC 5321 maximum email length
username     VARCHAR(50)   -- typical username max
description  TEXT          -- no arbitrary limit needed

-- DATE/TIME: always use TIMESTAMPTZ for events
created_at   TIMESTAMPTZ   -- '2024-03-15 14:30:00+05:30' ✅
created_at   DATETIME      -- loses timezone info — dangerous! ❌
birth_date   DATE          -- no time needed for birthdays

-- INTERVAL for date arithmetic
SELECT hire_date + INTERVAL '90 days' AS probation_ends FROM employees;

-- BOOLEAN — semantic clarity over TINYINT(1)
is_active   BOOLEAN DEFAULT TRUE
is_verified BOOLEAN DEFAULT FALSE

-- JSONB for flexible/semi-structured data (PostgreSQL)
CREATE TABLE products (
    product_id SERIAL      PRIMARY KEY,
    name       VARCHAR(200),
    metadata   JSONB        -- {"color":"red","size":"XL","weight":0.5}
);
SELECT name, metadata->>'color' AS color
FROM   products
WHERE  metadata @> '{"size":"XL"}';  -- JSON containment query
